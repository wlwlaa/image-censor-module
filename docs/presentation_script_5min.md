# Сценарий защиты: 5 минут

## 0:00–0:30 Проблема

Банк подключает генеративные модели для изображений: маркетинг, внутренние сервисы, обработка клиентских материалов. Но есть принципиальный риск: генератор может вернуть опасный контент, персональные данные или артефакт, который был подменён после проверки.

Поэтому наша задача — не просто определить риск, а не допустить выдачу непроверенного результата.

## 0:30–1:00 Почему встроенного фильтра мало

Встроенный фильтр зависит от конкретного генератора и его поставщика. При смене модели правила и уровень контроля меняются. А если detector недоступен, система не должна молча пропускать результат.

Генератор считается недоверенным компонентом.

## 1:00–1:40 Идея: независимый AI Security Gateway

Мы не строим ещё один классификатор изображений.

Мы строим независимый AI Security Gateway между генератором и пользователем банка. Он принимает вход, запускает проверки, помещает результат генерации в карантин, применяет policy и только после решения `ALLOW` создаёт release.

Ни один артефакт не может попасть пользователю без независимой проверки и подписанного паспорта безопасности.

## 1:40–2:30 Архитектура

Путь артефакта короткий и обязательный.

Сначала проверяем prompt и, для `img2img`, входное изображение. Затем результат генератора попадает не пользователю, а в quarantine. После этого проверяем выход, применяем policy engine и работаем по принципу fail closed: ошибка safety-компонента означает запрет выдачи.

Для разрешённого результата выпускается паспорт безопасности. В нём есть artifact ID, SHA-256, версия policy, версии detector-адаптеров, timestamp и подпись. При скачивании система снова проверяет подпись, привязку к artifact ID и hash файла.

## 2:30–3:30 Демо сценарии

Первый сценарий — безопасное изображение. Видим `ALLOW`, подписанный паспорт и verified download.

Второй — PII на входе `img2img`. Gateway блокирует запрос до генератора. Это важно: чувствительные данные не должны уходить дальше по цепочке.

Третий — detector failure. Safety-компонент недоступен, и Gateway возвращает `BLOCK`. Это fail closed.

Четвёртый — подмена разрешённого файла после проверки. При скачивании hash и подпись не сходятся, поэтому система отвечает `HTTP 409`.

Есть и пятый сценарий: опасный результат блокируется после quarantine и не попадает в release.

## 3:30–4:20 Аналоги и отличие

Мы рассматриваем ShieldGemma 2, Azure AI Content Safety, AWS Bedrock Guardrails и OpenAI Moderation как полезные компоненты экосистемы. UnsafeBench полезен для оценки detector-слоя.

Готовые moderation API полезны как detector, но не заменяют банковский enforcement-контур.

У AWS Bedrock Guardrails уже есть развитые механизмы фильтрации и блокировки. Наш тезис не в том, что аналоги слабые. Наш тезис в другом: банку нужен независимый маршрут допуска конкретного артефакта с quarantine, собственной policy, паспортом, проверкой целостности при выдаче и audit.

## 4:20–5:00 Метрики, ограничения, roadmap

MVP доказывает архитектуру и пять enforcement-сценариев. Мы честно не заявляем production ML quality: сейчас используются детерминированные demo-адаптеры, локальное хранилище и HMAC-секрет из окружения.

Следующий шаг пилота — подключить отдельно валидированные detector-адаптеры, OCR/DLP и банковскую таксономию. Затем — private object storage, tenant-aware access, KMS/HSM, WORM audit и интеграция с SIEM/SOC.

Итог: GenSecOps делает безопасность GenAI-артефакта не обещанием модели, а проверяемым банковским процессом выдачи.
